# Trae 使用分享：基于 VLA 雷达仿真项目的开发实践

## 一、项目背景

VLA（Very Large Array）是一个基于大模型的雷达组网仿真与智能辅助决策系统。项目分为三个主要模块：
- **M1：意图驱动航迹生成器** - 生成目标的真实轨迹数据
- **M2：高保真传感器仿真引擎** - 模拟雷达对目标的观测
- **M4：态势可视化模块** - 可视化展示轨迹和雷达观测数据

这个项目是使用 **Trae Space** 功能进行开发的典型案例，完整展示了从需求到交付的全流程。

## 二、Trae Space 核心功能实践

### 2.1 规范的项目开发文档体系

Trae 提供了完整的文档管理机制，在 `.trae/specs/` 目录下，每个模块都有一套独立的文档：

```
.trae/specs/
├── implement-m1-trajectory-generator/
│   ├── spec.md          # 规格说明文档
│   ├── tasks.md         # 任务拆解文档
│   └── checklist.md     # 验收清单文档
├── implement-m2-sensor-simulation/
│   ├── spec.md
│   ├── tasks.md
│   └── checklist.md
└── implement-m4-situation-visualization/
    ├── spec.md
    ├── tasks.md
    └── checklist.md
```

#### 2.1.1 spec.md（规格说明文档）

**作用**：详细描述模块的功能需求、技术方案、接口规范等。

**M1 模块 spec.md 的核心内容示例**：
- 模块定位与功能概述
- 数据流转与处理步骤
- 输入输出格式规范
- 核心算法说明（如：航路展开与平滑转弯算法）
- 工程落地建议

```markdown
## 为什么需要这个模块
M1是整个雷达组网仿真系统的数据源头，扮演"战场导演"角色。

## 变更内容
- 实现飞行简令文本解析模块（正则表达式提取）
- 实现本地知识库（地理映射表、机型性能表、航路点库）
- 实现航路展开与平滑转弯算法
- ...
```

#### 2.1.2 tasks.md（任务拆解文档）

**作用**：将模块开发拆解为可执行的子任务，并标注任务依赖关系。

**M1 模块 tasks.md 示例**：
```markdown
# 任务列表

- [ ] 任务1：项目结构搭建与基础配置
  - [ ] 子任务1.1：创建M1模块目录结构
  - [ ] 子任务1.2：创建requirements.txt依赖文件
  - [ ] 子任务1.3：创建模块入口文件和配置文件

- [ ] 任务2：本地知识库实现
  - [ ] 子任务2.1：实现地理映射表数据结构和查询接口
  - ...

# 任务依赖关系
- [任务3] 依赖于 [任务2]
- [任务4] 依赖于 [任务2], [任务3]
```

#### 2.1.3 checklist.md（验收清单文档）

**作用**：列出模块交付前需要验证的功能点。

## 三、面对未知领域：如何补充专业知识

在开发 M1 模块时，我们遇到了一个挑战：**航路规划需要专业的民航知识，大模型可能会"胡编乱造"**。

### 3.1 问题分析

初期方案过度依赖在线大模型生成航路，存在以下问题：
- 成本高且不稳定
- 生成的航路不够专业
- 缺乏真实数据支撑

### 3.2 解决方案：从"AI 主导"到"数据主导 + AI 辅助"

我们采取了以下步骤：

#### 第一步：搜集真实专业资料
从官方渠道获取了 **EAIP（Aeronautical Information Publication）民航资料**：
```
2026 Nr.04/
├── GEN_2.4_Location indicators.pdf      # 地名代码(完整机场列表)
├── ENR_4.4_Name-code designators.pdf    # 重要点航路点(735个)
├── AD_1.3_Index to aerodromes.pdf       # 机场索引
└── [A-Z]{4}.pdf                         # 具体机场数据文件(22个)
```

#### 第二步：数据提取与分析
编写脚本从 PDF 中提取结构化数据，并生成 `EAIP数据分析报告.md`：
- 成功提取 22 个机场的精确坐标
- 提取 735 个标准航路点
- 构建了本地知识库

#### 第三步：重构系统架构

**新的三级降级机制**：
```
优先顺序:
1. ✅ EAIP标准航路数据库（首选）
2. ✅ 大模型AI辅助（只在无标准航路时）
3. 降级: 简单大圆航线（兜底方案）
```

#### 第四步：将知识喂给系统（而不是只靠大模型）

在代码中实现了：
```python
# 本地知识库模块
load_eaip_airports()      # 加载EAIP机场数据
load_eaip_waypoints()     # 加载EAIP航路点数据
find_standard_route()     # 查找标准航路
```

## 四、Trae 使用心得总结

### 4.1 核心优势

1. **文档驱动开发**
   - 先写 spec，再写代码，避免返工
   - 任务拆解清晰，进度可控
   - 验收标准明确，交付质量有保障

2. **Space 功能的价值**
   - 每个模块有独立的开发空间
   - 文档和代码放在一起，上下文完整
   - 便于团队协作和知识传承

3. **灵活应对未知领域**
   - 不盲目依赖大模型
   - 结合真实数据，构建领域知识库
   - 大模型作为辅助，而非主导

### 4.2 最佳实践建议

#### 实践1：建立"文档三剑客"体系
- **spec.md**：回答"做什么"和"怎么做"
- **tasks.md**：回答"拆成哪些任务"
- **checklist.md/验收文档**：回答"怎么算做完"

#### 实践2：面对不熟悉的领域
1. **先调研**：搜集官方资料、专业文档
2. **再构建知识库**：将资料结构化，存入本地
3. **最后用大模型**：让大模型在知识库基础上工作，而非凭空创造
4. **保留降级机制**：当 AI 失败时，有兜底方案

#### 实践3：项目结构建议
```
项目根目录/
├── .trae/specs/          # Trae Space 文档
├── 方案文档/              # 详细技术方案
├── [模块1]/              # 模块代码
├── [模块2]/              # 模块代码
├── output/               # 输出数据
└── 各种验收文档.md        # 验收标准
```

## 六、总结

Trae Space 不仅仅是一个代码编辑工具，更是一套**完整的开发方法论**：

1. **文档先行**：用 spec、tasks、checklist 锚定开发方向
2. **任务拆解**：将复杂项目拆分为可管理的小任务
3. **质量保障**：通过验收文档确保交付质量
4. **知识沉淀**：将专业知识固化到代码库中，而非只存在于大模型里

对于需要专业领域知识的项目，关键是：**不要让大模型"猜"，而是给大模型"喂"真实数据，让它在正确的基础上工作**。
