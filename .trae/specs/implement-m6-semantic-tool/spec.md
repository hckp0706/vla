# M6：数据语义化转换工具链 规格说明

## 背景与目的
M2 融合航迹以高频时序JSON输出，包含经纬度、速度、航向、SNR、敌我属性等大量数值字段。直接将原始数据输入大模型，会因上下文过长、数值密集导致大模型无法有效理解态势。需要一个"翻译官"工具，将数字航迹压缩为结构化的自然语言战报，使大模型能高效理解并推理。

## 变更内容
- 新增 `m6_semantic_tool` 模块，实现 M2 融合航迹到自然语言描述的转换
- 从 M2 的 `FusedNetworkTracks`（综合融合航迹）中提取关键态势特征
- 内置地理知识库（52个中国主要城市、8个重点区域、30个机场坐标），用于坐标→地名反查
- 计算航迹相对于 M2 各传感器的最近距离
- 推断飞行特征（低空突防、高空侦察、超音速飞行等11种标签）
- 实现航迹按航向分段描述（>15° 转折触发新段）
- 实现高度/速度统计自适应骤变检测（mean+2σ + 绝对值过滤）
- 实现城市途经检测（100km 阈值内按时间插入描述）
- 实现雷达站名称映射（输出"青岛雷达"而非"RADAR_01"）
- 实现机场级起降识别（5km内优先匹配机场，降级链：机场→城市→区域→坐标）
- 生成针对每条航迹的多段式自然语言战报文本
- 提供命令行入口和 Python API
- 通过 Flask API（`/api/m6/convert`）与 M4 可视化界面集成
- 自动保存输出结果到 `output/m6_semantic_tool/`

## 影响范围
- 涉及模块：M6 数据语义化转换（新增模块）
- 依赖数据：
  - `m2_sensor_simulation.models`（TrackPoint, NetworkTracks）
  - `m2_sensor_simulation.models_ext`（FusedTrackPoint, FusedNetworkTracks）
  - `m2_sensor_simulation.radar_config.json`（雷达阵地坐标）
  - `m2_sensor_simulation.sensor_config.json`（SSR/IFF/ESM等传感器坐标）
- 无 M1 依赖，仅加载 M2 综合航迹

## 新增需求

### 需求：融合航迹加载与解析
系统 SHALL 仅加载 M2 输出的综合融合航迹文件：
- 加载 `fused_comprehensive.json`（综合融合航迹）
- 解析为按 track_id 分组的独立航迹时间序列

#### 场景：加载综合融合航迹
- **当** 用户传入 M2 综合融合航迹 JSON 文件路径
- **则** 系统解析为航迹列表，每条航迹按 track_id 分组为时间序列

### 需求：内置地理知识库
系统 SHALL 内置中国主要城市坐标和重点区域经纬度范围数据，用于将经纬度坐标反查为可读地名：

1. **主要城市坐标表**（用于坐标→城市名反查）：
   - 包含中国省会城市、直辖市、特别行政区及重点城市的经纬度坐标
   - 反查时取距离最近的城市（阈值 100km 内），如"北京附近"、"上海附近"
   - 超出阈值则降级为区域描述

2. **重点区域经纬度范围**（用于坐标→区域名反查）：
   - 渤海海域、黄海海域、东海海域、南海海域
   - 台湾海峡、琼州海峡
   - 朝鲜半岛、日本列岛周边
   - 反查时判断坐标是否落在某区域经纬度范围内

3. **机场坐标表**（用于起降点识别）：
   - 包含中国主要机场（民用+军用）的 ICAO 代码、名称、经纬度
   - 用于识别起飞/降落地点，如"青岛胶东国际机场附近"

#### 场景：坐标反查为城市名
- **当** 航迹起始坐标为 (120.38, 36.07)
- **则** 反查结果为"青岛附近"

#### 场景：坐标反查为海域
- **当** 航迹经过坐标落在黄海海域范围内
- **则** 途经描述中标注"黄海海域上空"

#### 场景：坐标反查为机场
- **当** 航迹起止坐标距某机场 < 5km
- **则** 起降点描述为该机场名称

#### 场景：坐标无法匹配任何已知地点
- **当** 坐标距最近城市 > 100km 且不在任何已知区域内
- **则** 降级描述为"坐标({lon},{lat})附近"

### 需求：航迹特征提取
系统 SHALL 从单条航迹的时序数据中提取以下关键特征：

1. **时间特征**：
   - 首次出现时间（起飞/进入探测范围）
   - 末次出现时间（降落/离开探测范围）
   - 总持续时间

2. **空间特征**：
   - 起始位置（首次观测经纬度 + 地理反查名称）
   - 终止位置（末次观测经纬度 + 地理反查名称）
   - 途经关键点（航向发生显著变化的位置，转向角度 > 30°，附地理反查名称）
   - 与各传感器的最近距离及对应时刻

3. **运动特征**：
   - 最大速度（km/h）
   - 平均速度（km/h）
   - 主要航向方向（八方位：东/南/西/北/东南/东北/西南/西北）
   - 高度范围（最低/最高/典型巡航高度）

4. **探测质量特征**：
   - 航迹质量等级分布（HIGH/MEDIUM/LOW/LOST 的占比）
   - 平均信噪比
   - 探测连续性（断续出现次数、最长连续探测时长）

5. **身份属性特征**：
   - 敌我属性（FRIEND/FOE/NEUTRAL/UNKNOWN）及置信度
   - SSR 信息（squawk 码、callsign、ICAO24）
   - 机载辐射源类型（来自 ELINT）
   - 通信活动（来自 COMINT）

#### 场景：提取完整航迹特征
- **当** 输入一条包含 N 个时序点的融合航迹
- **则** 系统输出结构化的 TrackFeatures 对象，包含上述所有特征

#### 场景：处理断续航迹
- **当** 航迹中间存在探测中断（quality=LOST 的时段）
- **则** 系统在飞行特点描述中标注"航迹断续出现"

### 需求：飞行特征推断
系统 SHALL 基于提取的特征推断飞行特点，输出语义化的飞行特征标签：

| 推断规则 | 输出标签 |
|---------|---------|
| 平均高度 < 1000m 且存在贴近飞行段 | 低空飞行 |
| 平均高度 > 12000m | 高空飞行 |
| 高度变化范围 < 500m | 平稳巡航 |
| 存在明显的爬升段和下降段 | 正常起降 |
| 最大速度 > 900 km/h | 高速飞行 |
| 最大速度 > 1500 km/h | 超音速飞行 |
| 平均速度 < 300 km/h | 低速飞行 |
| 航向变化 > 90° 且存在多段不同航向 | 航线机动 |
| 敌我属性为 FOE 且低空飞行 | 低空突防 |
| 敌我属性为 FOE 且高空高速 | 高空侦察 |

#### 场景：推断低空突防特征
- **当** 航迹平均高度 < 1000m，敌我属性为 FOE
- **则** 飞行特点中包含"低空突防"

### 需求：自然语言描述生成
系统 SHALL 将航迹特征转换为流畅的自然语言描述文本，格式如下：

```
目标批号{target_id}，{敌我属性描述}，于{起始时间}从{起飞地点}起飞，
在{途经时间}途经{途经地点}，整体飞行特点为{飞行特征}，
最大速度{最大速度}km/h，航向朝{航向方向}，
飞行高度{高度描述}，距我{最近传感器}最近距离为{最近距离}km，
于{结束时间}降落于{降落地点}。{补充信息}
```

各字段说明：
- `{敌我属性描述}`: 如"判断为敌方目标（置信度0.85）"或"友方目标"
- `{起飞地点}`: 通过地理知识库反查坐标对应的城市名/机场名/区域名
- `{途经地点}`: 航向转折点经地理知识库反查后的地名
- `{飞行特征}`: 飞行特征标签组合，如"低空突防、高速飞行"
- `{航向方向}`: 主要航向的八方位描述，如"东南方向"
- `{高度描述}`: "高度约{平均高度}米"或"低空飞行（{最低高度}-{最高高度}米）"
- `{最近传感器}`: 距离最近的传感器名称
- `{降落地点}`: 通过地理知识库反查坐标对应的城市名/机场名/区域名
- `{补充信息}`: SSR callsign、辐射源类型、通信活动等

#### 场景：生成完整自然语言描述
- **当** 输入一条航迹的特征数据
- **则** 系统输出一段完整、流畅的自然语言描述文本

#### 场景：坐标反查为城市名后生成描述
- **当** 航迹起始坐标反查为"青岛附近"
- **则** 描述中起飞地点显示为"青岛附近"

### 需求：航迹分段描述（按航向变化）
系统 SHALL 将航迹按航向显著变化（>15°）自动切分为多个飞行段，每段独立描述：

#### 场景：按航向自动分段
- **当** 航迹在某点的航向变化超过 15°
- **则** 该点产生新的飞行段，每段输出格式为"时间区间，朝方位角xxx°向xx方向飞行，高度从xx米变化至xx米"

#### 场景：多段航迹描述
- **当** 航迹包含 7 个航向转折段
- **则** 自然语言描述中包含 7 段飞行路径描述，按时间顺序排列

### 需求：高度/速度骤变检测（统计自适应）
系统 SHALL 基于统计自适应方法检测航迹中的高度和速度异常变化：

- 计算全航迹高度/速度的标准差 σ
- 当相邻点变化量超过 mean + 2σ 阈值时标记为异常
- 高度变化必须 > 200m 绝对值，速度变化必须 > 50km/h 绝对值（防止误报）
- 异常事件按时间顺序插入到对应飞行段描述中

#### 场景：检测高度骤变
- **当** 航迹在某点高度从 5000m 骤降至 2000m（变化超过统计阈值且 > 200m）
- **则** 描述中插入"xx时刻高度发生骤变，从5000米变化到2000米"

#### 场景：忽略微小异常
- **当** 高度变化仅 50m（不满足 >200m 绝对值条件）
- **则** 不标记为高度骤变

### 需求：城市途经检测
系统 SHALL 检测航迹每个时刻与已知城市的最近距离，将途经（距离 < 100km）时刻按时间插入到对应飞行段描述中：

#### 场景：检测城市途经
- **当** 航迹某时刻距"北京"仅 40.1km
- **则** 该段描述中插入"途经北京上空（距离40.1km）"

### 需求：雷达站名称映射
系统 SHALL 在自然语言描述中使用雷达站的真实名称（如"青岛雷达"、"上海雷达"）替代传感器ID（如"RADAR_01"）：

#### 场景：雷达站名称显示
- **当** 计算航迹距传感器最近距离时
- **则** 描述输出"距我青岛雷达最近距离为xxxkm"，而非"RADAR_01"

### 需求：机场级起降识别
系统 SHALL 在起降点识别时优先匹配机场（5km 阈值），其次城市（100km 阈值），最后区域：

#### 场景：识别北京首都机场起降
- **当** 航迹起止坐标距北京首都国际机场 < 5km
- **则** 起飞/降落地点显示为"北京首都国际机场附近"

#### 场景：机场非最近时降级为城市
- **当** 航迹坐标距最近机场 > 5km 但距城市 < 100km
- **则** 起飞/降落地点显示为"北京附近"

### 需求：中文时间格式
系统 SHALL 使用中文自然语言时间格式：

#### 场景：中文时间格式输出
- **当** 航迹起始时间为 14:00:00
- **则** 输出"14时00分00秒"而非"14:00:00"

### 需求：批量航迹处理与全局态势汇总
系统 SHALL 支持一次处理整个 FusedNetworkTracks，为每条独立航迹生成自然语言描述：

#### 场景：处理多目标场景
- **当** 输入包含多条航迹（多个 track_id）的 FusedNetworkTracks
- **则** 系统为每条航迹独立生成自然语言描述，输出结构化的 SituationReport 对象

#### 场景：全局态势汇总
- **当** 完成所有航迹的描述生成后
- **则** 系统生成一段全局态势摘要，包含：目标总数、敌方目标数、最近目标及距离、最高威胁等级

### 需求：M4 界面集成（Flask API）
系统 SHALL 通过 Flask 服务器对外暴露 REST API，供 M4 态势可视化界面调用：

#### 场景：M4 前端调用语义转化
- **当** M4 前端 POST 请求 `/api/m6/convert`，携带 M2 融合航迹数据
- **则** 返回 JSON，包含 success、text、summary、saved_json、saved_txt 字段

#### 场景：转化结果自动保存
- **当** 每次 API 调用完成语义转化
- **则** 结果自动保存到 `output/m6_semantic_tool/` 目录（JSON + TXT 双格式）

### 需求：增强的飞行特征推断
系统 SHALL 支持更丰富的飞行特征标签推断规则：

| 推断规则 | 输出标签 |
|---------|---------|
| 敌我属性为 FOE 且低空飞行 | 低空突防 |
| 敌我属性为 FOE 且高空高速飞行 | 高空侦察 |
| 存在探测间隙 | 航迹断续 |

#### 场景：FOE 低空判定为低空突防
- **当** 航迹敌我属性为 FOE 且平均高度 < 1000m
- **则** 飞行特征标签为"低空突防"而非单独的"低空飞行"

### 需求：传感器距离计算
系统 SHALL 计算每条航迹相对于 M2 各传感器的最近距离：

- 从 radar_config.json 和 sensor_config.json 读取所有传感器阵地坐标
- 包括雷达（RADAR_01~05）、SSR、IFF、ESM、ELINT、COMINT 站点
- 对航迹的每个时序点，计算与各传感器的 Haversine 距离
- 记录最近距离及对应时刻

#### 场景：计算最近传感器距离
- **当** 航迹包含 N 个时序点，系统有 M 个传感器
- **则** 输出每个传感器对应的最近距离（km）和最近时刻

### 需求：命令行接口
系统 SHALL 提供 `python -m m6_semantic_tool` 命令行入口：

```
python -m m6_semantic_tool -i <M2综合融合航迹文件> [-o <输出文件>] [--format text/json]
```

- `-i/--input`: 必选，M2 输出的综合融合航迹 JSON 文件路径
- `-o/--output`: 可选，输出文件路径（默认输出到控制台）
- `--format`: 可选，输出格式（text/json），默认 text

#### 场景：命令行运行
- **当** 用户执行 `python -m m6_semantic_tool -i fused_comprehensive.json`
- **则** 系统输出每条航迹的自然语言描述到控制台

### 需求：Python API 接口
系统 SHALL 提供简洁的 Python API 供 M3（认知中枢）直接调用：

```python
from m6_semantic_tool import SemanticConverter

converter = SemanticConverter(
    radar_config_path="m2_sensor_simulation/radar_config.json",
    sensor_config_path="m2_sensor_simulation/sensor_config.json"
)

# 从文件加载并转换
report = converter.convert_from_file(
    fused_track_path="output/m2_tracks/fused_comprehensive.json"
)

# 直接传入 FusedNetworkTracks 对象
report = converter.convert(fused_tracks)

# 获取结果
print(report.to_text())     # 全部自然语言文本
print(report.to_dict())     # 结构化字典
print(report.summary)       # 全局态势摘要
```

#### 场景：Python API 调用
- **当** M3 模块调用 `converter.convert(fused_tracks)`
- **则** 返回 SituationReport 对象，包含每条航迹的自然语言描述和全局摘要
