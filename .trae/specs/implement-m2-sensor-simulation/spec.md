# M2：高保真传感器仿真引擎 技术方案 (MVP 阶段)

## 1. 模块定位与功能概述

M2 是整个系统的**“战场迷雾制造机”**。如果说 M1 是全知全能的上帝，M2 就是蒙在上帝眼睛上的一层带有漏洞的纱布。它接收 M1 的完美真值，严格遵循电磁波传播物理规律与雷达信号处理逻辑，剥落那些由于距离、地球曲率、地形遮挡或杂波抑制而“看不见”的目标，并给剩下的目标加上量测噪声与滤波平滑，最终输出组网融合后的目标航迹，供指挥所（M3/M6）使用。

## 3. 核心物理与信号处理流程（按 1Hz Tick 驱动）

在每个 1 秒的时间片内，M2 遍历所有 5 部雷达，针对 M1 传来的每一个目标真值，严格按顺序过以下“五关”物理计算：

### 第一关：视距与地形遮挡判定（严格物理阻断）
雷达波无法拐弯，这一关决定目标在物理上是否“可见”。
1.  **雷达视距方程计算**：受地球曲率影响，雷达看不了无限远。严格计算：
    $$R_{视距} = 4.12 \times (\sqrt{H_{雷达}} + \sqrt{H_{目标}})$$
    若目标真实斜距 $R > R_{视距}$，**直接丢弃**（低空突防防线的核心物理依据）。
2.  **地形遮蔽判断（可选接入）**：若加载 DEM 高程数据，插值计算雷达与目标连线上的最高障碍物。若被挡住，**直接丢弃**。

### 第二关：波束驻留判定（模拟机械扫描）
雷达不是全天候盯着一个方向，只有波束扫过瞬间才有回波。
1.  根据当前系统时钟与雷达的“扫描周期”，算出当前时刻雷达波束的物理指向角 $\theta_{扫描}$。
2.  计算目标相对雷达的真实方位角 $\theta_{目标}$。
3.  若 $|\theta_{目标} - \theta_{扫描}| > \text{波束宽度}/2$，说明波束没扫到，**直接丢弃**（这就解释了为什么 1Hz 的真值，在雷达屏幕上是断续出现的）。

### 第三关：雷达方程与基础 SNR 计算（能量衰减计算）
针对通过前两关的目标，计算回波能量大小：
1.  提取 M1 传来的目标当前真实 RCS 值（`rcs_dbsm`）。
2.  代入简化的雷达方程（以 dB 形式加减）：
    $$SNR_{基础} = P_t + G_t + G_r + 20\log_{10}(\lambda) + RCS - 40\log_{10}(R) - Losses - KTB$$
    *(核心体现 RCS 越小、距离 R 越远，SNR 呈指数级衰减的物理规律)*

### 第四关：MTI/MTD 杂波抑制与 CFAR 检测（信号处理等效）
这一关决定微弱回波能否从杂波背景中被提取出来。
1.  **计算径向速度 ($V_r$)**：根据目标的真值速度、航向与相对雷达的几何关系，计算出目标直奔/背离雷达而来的径向速度。
2.  **MTI/MTD 等效滤除**：真实雷达的多普勒滤波器无法滤除与地物杂波同频的低速目标。
    *   判定逻辑：如果 $|V_r| < 30 \text{ m/s}$（可配置杂波抑制盲区），认为目标陷入强地杂波。
    *   动作：强行压低信噪比：$SNR_{实际} = SNR_{基础} - 20 \text{ dB}$。
3.  **CFAR 门限判决**：设定等效恒虚警检测门限（如 $12 \text{ dB}$）。若 $SNR_{实际} < 12 \text{ dB}$，**直接丢弃**（漏检）；否则判定为“发现点迹”。

### 第五关：量测噪声注入与卡尔曼滤波航迹生成
被检测到的点迹不能直接丢给大模型，否则会乱跳，必须经过航迹处理。
1.  **注入噪声**：根据当前的 $SNR_{实际}$，计算测距/测角误差标准差 $\sigma$。在目标的真实极坐标上叠加高斯白噪声 $\mathcal{N}(0, \sigma^2)$，反算回带有误差的观测经纬度/高度。
2.  **点迹关联**：拿着带噪观测值，去该雷达维护的“已知航迹池”里找距离最近的（最近邻算法）。
3.  **卡尔曼滤波平滑**：使用标准的恒定速度（CV）模型卡尔曼滤波，用带噪观测值更新航迹状态估计，输出平滑后的位置和速度。

## 4. 接口规范与数据字典

### 4.1 输入接口 (来自 M1)
直接接收 M1 的 1Hz `TruthFrame` JSON。重点关注字段：`target_id`, `lon`, `lat`, `alt_m`, `speed_ms`, `heading_deg`, `rcs_dbsm`。

### 4.2 输出接口 (传递给 M6 语义化模块)
M2 整合 5 部雷达的局部航迹，输出系统级组网观测航迹 JSON：

```json
{
  "frame_time": "2026-04-20T10:05:01Z",
  "network_tracks": [
    {
      "track_id": "T01",           // 系统航迹批号 (MVP阶段直接透传M1的ID便于调试对比)
      "source_radars": ["RADAR_01", "RADAR_04"], // 哪些雷达当前波束扫到了它
      "obs_lon": 119.8215,         // 滤波平滑后的观测经纬度 (带系统误差)
      "obs_lat": 32.1095,
      "obs_alt_m": 5150.0,
      "obs_speed_ms": 245.0,       // 滤波平滑后的观测速度
      "obs_heading_deg": 245.0,    // 滤波平滑后的观测航向
      "track_quality": "HIGH",     // 综合质量评估 (基于参与雷达数量和平均SNR)
      "snr_avg_db": 18.5           // 当前周期的平均信噪比
    }
    // 注意：如果某目标在这一秒被所有雷达漏检（比如正处于扫描间隙或被地形遮挡），此列表中将不存在该 ID
  ]
}
```

## 5. 异常处理与工程防呆

1.  **ID 透传原则**：真实对抗中雷达不知道目标真实编号。但 MVP 阶段为了在 M4 界面上直观对比“上帝视角(M1)”与“雷达视角(M2)”的差异，**强制将 M1 的 `target_id` 透传**，绝对禁止在 M2 内部重新编批号，否则后续调试将陷入灾难。
2.  **航迹生死管理**：当目标连续 3 个扫描周期未被任何雷达发现（可能掉线或飞出视距），必须在输出列表中**主动删除**该 `track_id`，不能像幽灵一样一直留在屏幕上。

## 6. 开发技术栈建议

*   **空间坐标转换**：经纬度高斯投影与极坐标互转，强推使用 `pysolar` 或基于 `pyproj` 库，纯手写极易产生累积偏航误差。
*   **滤波算法**：**严禁手写矩阵求逆**！直接 `pip install filterpy`，调用 `KalmanFilter` 类，5 行代码实现标准的二维/三维 CV 模型。
*   **并发计算**：5 部雷达对几十个目标的计算是相互独立的，使用 Python `concurrent.futures.ThreadPoolExecutor` 开 5 个线程并行跑，确保 1 秒内完成一帧的计算不卡顿。

## 7. 架构设计考量：关于“功能级等效仿真”的理念说明

在阅读本方案时，工程师可能会疑惑：*“真实的 MTI/MTD 是对脉冲串做快速傅里叶变换(FFT)，真实的 CFAR 是计算滑窗内的杂波均值，为什么这里只是一堆 if-else 判断？”*

这里必须明确 M2 的工程定位：**我们做的是“功能级/信号级交界面的白盒仿真”，而不是“底层 IQ 数据级的黑盒仿真”。**

如果在仿真引擎中去生成中频信号、加噪声、做脉冲压缩、跑 FFT 滤波器组，不仅代码量庞大，且 1 秒内根本算不完几十部雷达的几百个目标。

本方案遵循的等效原则是：**保留物理规律的“因果输入输出”，跳过底层的“数学变换细节”。**
*   我们不算 FFT，但我们通过提取**径向速度**，等效模拟了 FFT 把目标从杂波中分离出来的**结果**。
*   我们不算滑窗均值，但我们通过设定**固定门限和 SNR 衰减**，等效模拟了杂波导致发现概率下降的**现象**。

这种“抓效果、舍过程”的降维策略，是军事仿真系统从实验室走向工程落地的必经之路。它确保了 M2 给到 M3（大模型认知中枢）的数据特征（断续、噪点、低空丢失），与真实雷达处理出来的特征在统计学上是高度一致的。
