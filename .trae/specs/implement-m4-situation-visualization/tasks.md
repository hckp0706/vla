# M4 多维态势可视化界面任务清单

## 1. 模块定位与核心目标

在当前的系统闭环与逻辑验证阶段，M4 定位为**"带标准地理政治边界的静态全量态势对比看板"**。

防空作战不是在白纸上画线，必须具备强烈的地理与政治空间语境。M4 现阶段的核心目标是：加载标准的地理底图要素（国境、省界、ADIZ 等），在此基础上，一次性渲染 M1 的真实航线与 M2 的雷达观测航迹。

通过静态对比底图约束下的灰色线（M1真值）与彩色线（M2观测）的差异，工程师可以直观判断航迹是否违规飞越国境、是否进入防空识别区、以及雷达网在不同地理条件下的漏警情况。现阶段**不涉及任何时间轴、动画播放或拖拽逻辑**。

## 2. 底图数据渲染规范（核心新增）

底图要素必须严格分层渲染，遵循"政治边界醒目、地理参考弱化、业务数据突出"的原则。所有静态底图数据统一采用标准 **GeoJSON** 格式加载。

### 2.1 政治与主权边界层（最高优先级，视觉最强）

- [x] 任务 2.1: 实现国境线渲染
  - 数据源：标准国家基础地理信息数据（如 1:400万 GeoJSON）
  - 渲染样式：深黄色或黑色实线，线宽 2px。不允许被任何业务线条遮挡

- [x] 任务 2.2: 实现领海线渲染
  - 数据源：包含领海基点及 12 海里领海线坐标的 GeoJSON
  - 渲染样式：浅蓝色半透明多边形填充，边界线为蓝色虚线（区分陆地实线）

- [x] 任务 2.3: 实现东海防空识别区（ADIZ）渲染
  - 业务意义：后续大模型（M3）进行威胁评估的核心地理触发条件
  - 数据源：根据官方公布的六边形折线坐标点，手动编写成的 GeoJSON Polygon
  - 渲染样式：**红色半透明多边形填充（透明度极低，如 10%）**，外围用红色虚线勾边。在多边形内部空白处可放置静态文字标签"东海防空识别区"

### 2.2 地理参考层（中等优先级，视觉弱化）

- [x] 任务 2.4: 实现省/直辖市区域边界渲染
  - 数据源：简化版的省级 GeoJSON（仅保留东部沿海相关省份即可，如鲁、苏、浙、沪等，减少面数）
  - 渲染样式：浅灰色细实线（1px），无填充色，绝对不能喧宾夺主

- [x] 任务 2.5: 实现主要城市渲染
  - 数据源：包含经纬度的点要素 GeoJSON（仅保留省会及沿海地级市，如济南、青岛、南京、上海、杭州、宁波）
  - 渲染样式：白色圆点 + 城市名称文本标签（无背景框）

- [x] 任务 2.6: 实现主要机场位置渲染
  - 数据源：军民合用及主要军用机场坐标点 GeoJSON（可从 M1 知识库中直接导出）
  - 渲染样式：黑色小飞机图标或专属符号，鼠标悬停显示机场名称（如"上海虹桥"、"宁波栎社"）

### 2.3 我方阵地要素层（业务基准）

- [x] 任务 2.7: 实现当前雷达部署位置渲染
  - 数据源：直接硬编码 M2 方案中定义的 5 部雷达坐标（青岛、上海、宁波、南京、合肥）
  - 渲染样式：军用雷达扇形图标（蓝色）。外围画出其 `Rmax` 对应的**浅蓝色半透明圆形探测包络**

## 3. 业务数据渲染规范（M1 与 M2 异构处理）

**前提约束**：不改动 M1（按目标聚合）和 M2（按时间帧聚合）的输出设计方案，由前端承担数据格式转换的职责。

### 3.1 解析与渲染 M1 真值轨迹

- [x] 任务 3.1: 实现 M1 真值轨迹解析
  - 前端处理：遍历 M1 JSON 中的每个 `target_id`，直接提取其完整的 `track_points` 数组中的 `[lon, lat, alt]`

- [x] 任务 3.2: 实现 M1 真值轨迹渲染
  - 渲染样式：半透明浅灰色折线（线宽 2px）。因为是全量一次性画出，它必定是一条连续平滑的完整曲线

### 3.2 解析与渲染 M2 观测航迹（带断点重构）

- [x] 任务 3.3: 实现 M2 观测航迹解析
  - 前端处理（内存展平）：由于 M2 是按秒输出的帧数组，前端需在内存中建立以 `track_id` 为 key 的 Map。遍历所有帧，将同一个 ID 零散出现的坐标**重新拼接到一个数组中**

- [x] 任务 3.4: 实现 M2 观测航迹断点重构
  - 保留断续特征：在拼接时，如果发现同一 ID 在时间上出现了跳跃（比如上一秒在 T1，下一秒直接跳到了 T3，说明 T2 秒被漏检），前端必须在坐标数组中**强行插入一个 `null` 断点**

- [x] 任务 3.5: 实现 M2 观测航迹渲染
  - 渲染样式：根据 `track_quality` 映射颜色（亮绿=高质、黄色=中质、红色=低质），线宽 3px。遇到 `null` 时 Cesium 会自动断开线段，在地图上形成**直观的断续虚线效果**

## 4. 界面布局设计

去除时间轴等复杂交互，界面分为左侧全幅地图与右侧极简控制面板：

- [x] 任务 4.1: 实现左侧全幅地图区域
  - 背景：省界灰线、城市名
  - 政治层：黄色国境线、蓝色领海区、红色东海防空识别区
  - 基准层：蓝色雷达包络圆、雷达图标
  - 业务层：灰色M1全量连续航线、彩色M2断续航迹

- [x] 任务 4.2: 实现右侧极简控制面板
  - 数据加载控制：[选择 M1 JSON]、[选择 M2 JSON]、[一键渲染态势]
  - 底图图层开关：[✓] ADIZ 识别区、[✓] 领海线、[✓] 省界边界
  - 轨迹图层开关：[✓] 显示 M1 真值、[✓] 显示 M2 观测
  - 要素信息框：(悬停图标显示名称)

## 5. 技术栈选型

- [x] 任务 5.1: 集成 Cesium.js 地图引擎
  - 地图引擎：**Cesium.js**。虽然本阶段只做二维俯视，但 Cesium 原生对 GeoJSON 的 `Polygon`（如领海、ADIZ填充区）和 `Polyline`（带断点的航迹线）支持极其完美，性能优异

- [x] 任务 5.2: 搭建前端框架
  - 前端框架：**原生 HTML + Vanilla JS**。无需 Vue/React，避免过度设计

- [x] 任务 5.3: 准备数据源
  - 国境线、省界：推荐使用阿里云 DataV 的公开 GeoJSON 数据源
  - 东海防空识别区：需由工程师查阅公开资料，手动整理出包含 6-8 个关键拐点经纬度的 JSON 数组

## 6. 交付标准与防呆

- [x] 任务 6.1: 政治底线防呆验证
  - 国境线必须完整显示，如果因为视野缩放导致国境线断裂或被业务线覆盖，视为前端渲染不合格

- [x] 任务 6.2: ADIZ 业务验证
  - 当加载的 M1 航迹穿越了红色的"东海防空识别区"多边形时，视觉上能清晰看到灰色轨迹进入红区，为后续 M3 大模型做"进入ADIZ"研判提供直观对照

- [x] 任务 6.3: 雷达包络约束验证
  - 能够直观看到部分 M2 的彩色断续航迹，严格被限制在浅蓝色的雷达圆形包络内部；如果有彩色线出现在包络外，说明 M2 的视距计算存在严重 Bug

- [x] 任务 6.4: 断点验证
  - M2 的彩色线条绝对不能是一条完美的平滑连线，必须肉眼可见由于漏检导致的断点、缺角现象
