# M4：多维态势可视化界面 技术方案

## 1. 模块定位与核心目标

在当前的系统闭环与逻辑验证阶段，M4 已从 **"带标准地理政治边界的静态全量态势对比看板"** 演进为 **"带语义转化能力的多维态势交互看板"**：

- **第一阶段**（静态全量展示版）：加载标准地理底图要素，一次性渲染 M1 真实航线与 M2 雷达观测航迹，通过静态对比直观判断传感器表现。
- **第二阶段**（Flask 服务器 + M6 语义转化集成）：新增 Flask 后端服务器，提供 `/api/m6/convert` 接口桥接 M6 语义化工具；前端新增「语义转化」Tab 页，用户可在加载 M2 数据后选择任意航迹，一键生成结构化自然语言战报并在界面上查看结果。

防空作战不是在白纸上画线，必须具备强烈的地理与政治空间语境。M4 的核心目标是：加载标准的地理底图要素（国境、省界、防空识别区等），在此基础上，一次性渲染 M1 的真实航线与 M2 的雷达观测航迹，并支持将融合航迹转换为大模型可理解的自然语言描述。

## 2. 底图数据渲染规范（核心新增）

底图要素必须严格分层渲染，遵循"政治边界醒目、地理参考弱化、业务数据突出"的原则。所有静态底图数据统一采用标准 **地理数据格式** 加载。

### 2.1 政治与主权边界层（最高优先级，视觉最强）
*   **国境线**：
    *   **数据源**：标准国家基础地理信息数据（如 1:400万地理数据格式）。
    *   **渲染样式**：深黄色或黑色实线，线宽 2像素。不允许被任何业务线条遮挡。
*   **领海线（海上边界）**：
    *   **数据源**：包含领海基点及 12 海里领海线坐标的地理数据格式。
    *   **渲染样式**：浅蓝色半透明多边形填充，边界线为蓝色虚线（区分陆地实线）。
*   **东海防空识别区**：
    *   **业务意义**：后续大模型（M3）进行威胁评估的核心地理触发条件。
    *   **数据源**：根据官方公布的六边形折线坐标点，手动编写成的地理数据多边形格式。
    *   **渲染样式**：**红色半透明多边形填充（透明度极低，如 10%）**，外围用红色虚线勾边。在多边形内部空白处可放置静态文字标签"东海防空识别区"。

### 2.2 地理参考层（中等优先级，视觉弱化）
*   **省/直辖市区域边界**：
    *   **数据源**：简化版的省级地理数据格式（仅保留东部沿海相关省份即可，如鲁、苏、浙、沪等，减少面数）。
    *   **渲染样式**：浅灰色细实线（1像素），无填充色，绝对不能喧宾夺主。
*   **主要城市**：
    *   **数据源**：包含经纬度的点要素地理数据格式（仅保留省会及沿海地级市，如济南、青岛、南京、上海、杭州、宁波）。
    *   **渲染样式**：白色圆点 + 城市名称文本标签（无背景框）。
*   **主要机场位置**：
    *   **数据源**：军民合用及主要军用机场坐标点地理数据格式（可从 M1 知识库中直接导出）。
    *   **渲染样式**：黑色小飞机图标或专属符号，鼠标悬停显示机场名称（如"上海虹桥"、"宁波栎社"）。

### 2.3 我方阵地要素层（业务基准）
*   **当前雷达部署位置**：
    *   **数据源**：直接硬编码 M2 方案中定义的 5 部雷达坐标（青岛、上海、宁波、南京、合肥）。
    *   **渲染样式**：军用雷达扇形图标（蓝色）。外围画出其**最大探测距离**对应的**浅蓝色半透明圆形探测包络**。

## 3. 业务数据渲染规范（M1 与 M2 异构处理）

**前提约束**：不改动 M1（按目标聚合）和 M2（按时间帧聚合）的输出设计方案，由前端承担数据格式转换的职责。

### 3.1 解析与渲染 M1 真值轨迹
*   **前端处理**：遍历 M1 数据文件中的每个**目标编号**，直接提取其完整的**轨迹点数组**中的**经度、纬度、高度**。
*   **渲染样式**：半透明浅灰色折线（线宽 2像素）。因为是全量一次性画出，它必定是一条连续平滑的完整曲线。

### 3.2 解析与渲染 M2 观测航迹（带断点重构）
*   **前端处理（内存展平）**：由于 M2 是按秒输出的帧数组，前端需在内存中建立以**航迹编号**为**键**的**映射**。遍历所有帧，将同一个编号零散出现的坐标**重新拼接到一个数组中**。
*   **保留断续特征**：在拼接时，如果发现同一编号在时间上出现了跳跃（比如上一秒在 T1，下一秒直接跳到了 T3，说明 T2 秒被漏检），前端必须在坐标数组中**强行插入一个空值断点**。
*   **渲染样式**：根据**航迹质量**映射颜色（亮绿=高质、黄色=中质、红色=低质），线宽 3像素。遇到空值时地图引擎会自动断开线段，在地图上形成**直观的断续虚线效果**。

## 5. 技术栈选型

*   **地图引擎**：**三维地球引擎**。虽然本阶段只做二维俯视，但该引擎原生对地理数据格式的**多边形**（如领海、防空识别区填充区）和**折线**（带断点的航迹线）支持极其完美，性能优异。
*   **前端框架**：**原生超文本标记语言 + 原生脚本语言**。无需使用框架，避免过度设计。
*   **数据源获取建议**：
    *   国境线、省界：推荐使用阿里云数据可视化平台的公开地理数据源。
    *   东海防空识别区：需由工程师查阅公开资料，手动整理出包含 6-8 个关键拐点经纬度的数据数组。

## 6. 交付标准与防呆

1.  **政治底线防呆**：国境线必须完整显示，如果因为视野缩放导致国境线断裂或被业务线覆盖，视为前端渲染不合格。
2.  **防空识别区业务验证**：当加载的 M1 航迹穿越了红色的"东海防空识别区"多边形时，视觉上能清晰看到灰色轨迹进入红区，为后续 M3 大模型做"进入防空识别区"研判提供直观对照。
3.  **雷达包络约束验证**：能够直观看到部分 M2 的彩色断续航迹，严格被限制在浅蓝色的雷达圆形包络内部；如果有彩色线出现在包络外，说明 M2 的视距计算存在严重缺陷。
4.  **断点验证**：M2 的彩色线条绝对不能是一条完美的平滑连线，必须肉眼可见由于漏检导致的断点、缺角现象。

## 7. 第二阶段新增：Flask 服务器 + M6 语义转化集成

### 7.1 背景与目标
原有的 M4 通过浏览器直接打开 `index.html` 运行，用户手动选择本地文件。第二阶段的重点是：(1) 提供本地 Flask 服务器统一管理前端静态文件和 API；(2) 集成 M6 语义化转换工具，让用户能在 M4 界面上直接选择 M2 航迹、一键生成自然语言战报并查看结果。

### 7.2 Flask 服务器（`server.py`）
- **功能**：提供静态文件服务 + REST API
- **启动方式**：`python m4_situation_visualization/server.py --port 8080`
- **API 端点**：
  - `GET /` — 返回 M4 前端页面
  - `POST /api/m6/convert` — 接收 M2 融合航迹 JSON，调用 M6 Python 引擎返回语义化结果
  - `POST /api/m6/save` — 手动保存语义转化结果到 `output/m6_semantic_tool/`
- **自动保存**：每次 `/api/m6/convert` 调用成功后，自动保存 JSON + TXT 双格式文件到 `output/m6_semantic_tool/`

### 7.3 语义转化 Tab 页
在右侧控制面板顶部新增 Tab 栏，支持两个页签切换：

**Tab 1 — 态势控制**（原有全部功能）：
- 数据加载控制（M1、M2 文件选择 + 渲染/清除按钮）
- 底图图层开关
- 传感器部署图层开关
- 轨迹图层开关
- 敌我识别/航路/传感器/航迹线段图例
- 要素信息框（悬停显示）

**Tab 2 — 语义转化**（新增）：
- 航迹选择下拉框：自动识别已加载的 M2 数据，按 `track_id` 分组，显示敌我身份、目标批号、航迹点数
- "执行语义转化"按钮：禁用状态直到选择有效航迹
- 态势战报概要区：显示全局态势摘要（目标总数、敌我方数量、最近目标、最高威胁等级）
- 自然语言战报区：完整的多段式飞行描述（分段航迹、城市途经、异常检测、飞行特征标签、雷达距离等）
- "保存结果到文件"按钮：手动保存当前结果
- 状态提示条：loading/success/error 三种状态

### 7.4 技术实现要点
- Tab 切换通过 JavaScript 操作 CSS class `active`，无页面刷新
- `m2TrackIdMap` 全局 Map 维护已加载的 M2 航迹分组，供语义转化 Tab 使用
- 语义转化通过 `fetch('/api/m6/convert')` 异步调用，支持配置 `track_id` 参数过滤单条航迹
- 控制面板宽度从 280px 调整为 300px
- 新增 CSS 组件：`.tab-bar`、`.tab-btn`、`.tab-content`、`.track-selector`、`.semantic-result`、`.summary-box`、`.convert-status`
