1. # M1：意图驱动航迹生成器 技术方案

   ## 1. 模块定位与功能概述

   在整个雷达组网仿真系统中，M1 扮演着**“战场导演”**的角色。它的核心任务是：接收指挥员下达的飞行任务指令，结合真实的地理环境、空域航路规则和飞机性能参数，推演出一条符合物理规律与航空逻辑的“上帝视角”飞行轨迹。

   M1 生成的数据是整个系统数据流的源头。它不关心雷达能不能看到目标，它只负责如实地模拟出飞机在每一秒（1Hz频率）的位置、速度、姿态以及电磁反射特征（RCS），并将这些数据打包向后传递给 M2（雷达仿真引擎）。

   ---

   ## 2. 数据流转与处理步骤

   对于刚接触本系统的工程师，可以将 M1 内部的工作流理解为一条“加工流水线”，主要分为四个步骤：

   1.  **文本解析（提取要素）**：将人类读得懂的指令文本，拆解为程序能识别的“批号、时间、地点、动作”等孤立要素。
   2.  **知识映射（补全信息）**：拿着解析出的“地名”和“机型”，去本地知识库查表，把地名变成经纬度，把机型变成速度、高度限制参数；**同时根据机型匹配对应的预设航路，将起降两点展开为详细的航路点序列**。
   3.  **物理推演（生成轨迹）**：根据航路点序列和时间要求，利用地球椭球模型计算大圆航线，并处理航路点之间的平滑转弯，生成每秒的经纬度、速度和高度。
   4.  **特征估算（计算RCS）**：根据飞机当前的飞行状态（如是否放起落架、是否在急转弯），估算出当前时刻的雷达散射截面积（RCS），一并写入最终的数据包。

   ---

   ## 3. 接口规范与数据字典

   为了保证 M1 生成的数据能够被下游 M2 毫无障碍地直接读取，必须严格统一数据的输入输出格式。

   ### 3.1 输入规范：半结构化飞行简令
   为了提高系统解析的稳定性和速度，M1 要求按照固定的模板（简令）填写：
   *   **格式模板**：`[批号] [机型] 于 [起飞时间] 从 [地点A] 起飞，[途径动作] [地点B] [途径时间]，[降落动作] [地点C] [降落时间]。`
   *   **输入示例**：`0001 民航客机 于 2026-04-20 09:20:30 从 北京大兴机场 起飞，途径 西安机场 10:20:00，降落 上海虹桥机场 12:40:00。`

### 3.3 功能扩展与技术改进

**1. 多段航线规划**
- 支持带途经点的多段航线规划，如北京-青岛-上海
- 自动拼接不同航段的航路点，生成完整的航线

**2. 知识库扩展**
- 扩展了中国国内、日本、韩国的机场信息
- 添加了更多导航点和航路，如 BAYAN、CHINA、DOLAN、SHAOL、SHANG 等
- 构建了更完善的航路网络，支持更多航线组合

**3. 算法改进**
- 实现了基于大圆航线的轨迹生成
- 包含爬升、巡航、下降阶段的高度变化
- 智能选择符合航空标准的航路

**4. 误差处理**
- 实现了路径点缺失时的降级处理，确保系统稳定性
- 优化了时间计算逻辑，确保飞行时间合理

   ### 3.3 最终输出格式 (M1 Output JSON)
   M1 按照 1Hz（每秒一次）的频率输出以下格式的数据，直接作为 M2 的入口参数：

   ```json
   {
     "target_id": "0001",
     "platform_type": "民航客机",
     "track_points": [
       {
         "time": "2026-04-20T09:20:30Z",
         "lon": 116.594000, 
         "lat": 39.509000, 
         "alt_m": 25.0,
         "speed_ms": 0.0, 
         "heading_deg": 180.0, 
         "vertical_rate_ms": 0.0,
         "rcs_dbsm": 13.0,
         "phase": "GROUND_TAKEOFF"
       },
       {
         "time": "2026-04-20T10:00:00Z",
         "lon": 112.500000, 
         "lat": 36.800000, 
         "alt_m": 10000.0,
         "speed_ms": 240.0, 
         "heading_deg": 195.0, 
         "vertical_rate_ms": 0.0,
         "rcs_dbsm": 5.2,
         "phase": "CRUISING"
       }
     ]
   }
   ```
   **字段释义**：
   *   `lon/lat/alt_m`：经纬度与海拔高度。
   *   `speed_ms`：地速（米/秒）。
   *   `heading_deg`：航向角（以真北为0度，顺时针）。
   *   `vertical_rate_ms`：垂直速率（正值为爬升，负值为下降，0为平飞）。
   *   `rcs_dbsm`：当前时刻的雷达散射截面积估算值。
   *   `phase`：当前飞行阶段枚举（如起飞、爬升、巡航、下降、降落）。

   ## 4. 核心算法与物理逻辑详解

   ### 4.1 文本解析逻辑（正则提取）
   程序通过 Python 的正则表达式来精准“抠”出所需信息，这种方式速度极快且绝不走样。

   ```python
   import re
   
   # 定义匹配规则，将简令拆解为多个命名组
   INTENT_PATTERN = re.compile(
       r"(?P<target_id>\d+)\s+(?P<platform_type>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+于\s+"
       r"(?P<takeoff_time>\d{4}-\d{2}-\d{2}\s+\d{2}:\d{2}:\d{2})\s+从\s+"
       r"(?P<loc_start>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+起飞"
       r"(?:，?\s*(?P<action_mid>[\u4e00-\u9fa5]+)\s+(?P<loc_mid>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+(?P<time_mid>\d{2}:\d{2}:\d{2}))?"
       r"(?:，?\s*(?P<action_end>[\u4e00-\u9fa5]+)\s+(?P<loc_end>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+(?P<time_end>\d{2}:\d{2}:\d{2}))?"
   )
   # 经过正则匹配后，程序会提取出中间字典，供下一步查表使用
   ```

   ### 4.2 航路展开与平滑转弯算法（避免“直线飞行”）
   这是体现仿真保真度的关键环节。如果只算起降两地的直线，轨迹会非常生硬，且航向会瞬间突变。

   1.  **航路展开**：识别到“民航客机”后，程序自动去【表3】匹配“京陕沪民航航路A”，将原本的“大兴 -> 虹桥”两个端点，展开为包含十余个中间导航点的完整序列。
   2.  **直飞段计算**：在两个相邻航路点之间，调用专业的地理空间库（如 `geographiclib`）计算大圆航线，正常进行 1Hz 的速度和经纬度插值。
   3.  **转弯提前量计算**：飞机不会到了航路点正上方才生硬转弯。程序会根据当前速度和标准转弯率（通常为3度/秒），计算出**开始转弯的起点**和**结束转弯的终点**。
   4.  **圆弧平滑插值**：在转弯区间内，程序以航路点为圆心画一段圆弧。在这几秒内，飞机的 `heading_deg`（航向）会**平滑、匀速地**从上一个航向过渡到下一个航向，彻底消除航向突变。

   ### 4.3 航迹物理推演逻辑（高度与速度）
   *   **宏观校验**：根据展开后的航路点序列计算总距离，除以总时间得出所需平均速度。若超出机型极限，按最大速度飞行并自动顺延到达时间（附加 Warning 日志）。
   *   **高度剖面**：模拟真实的“起飞拉起 -> 爬升 -> 巡航平飞 -> 下降 -> 落地”曲线。根据知识库的爬升/下降率，结合距离目标的远近，自动计算 `vertical_rate_ms` 的变化。

   ### 4.4 动态 RCS 估算逻辑
   由于 M1 不知道雷达具体部署在哪里，采用基于“飞行姿态”的经验估算法则：
   *   **巡航态**：平飞时取“机头RCS”作为基准（例如民航为 5.0 dBsm）。
   *   **起降态增损**：起飞和降落阶段放下起落架，RCS = 基准值 + 起落架增量（如 5.0 + 8.0 = 13.0 dBsm）。
   *   **转弯态增损**：如果在执行 4.2 中的“圆弧平滑转弯”，说明侧面暴露。程序会将 RCS 从“机头值”向更高的“侧面值”进行线性插值过渡。

   ## 5. 异常处理与防呆机制

   1.  **地名模糊匹配**：简令写“大兴”，知识库叫“北京大兴机场”，程序会自动计算字符串相似度进行模糊匹配修正，不直接报错。
   2.  **地形防穿透**：若接入高程地形数据（DEM），每秒生成高度点时校验，若低于地表安全高度，强制拉平，防止出现“飞机钻地”。
   3.  **超时柔性处理**：遇到物理上飞不到的任务，不抛出异常中断仿真，而是按极限性能飞，并记录警告日志。

   ## 6. 架构设计考量：关于自然语言解析与大模型的抉择

   在阅读本方案时，工程师可能会有疑问：*“现在大模型（LLM）那么火，为什么文本解析（步骤1）不用大模型来做？”*

   这是一个关键的工程取舍，主要原因如下：

   1.  **底层仿真的容错率为零**：大模型具有“创造性”，偶尔会把 `time` 写成 `Time`，或漏掉标点导致 JSON 格式错误。对于以 1Hz 严格推演的物理引擎，一次解析失败就会导致整个推演流崩溃。
   2.  **系统对实时性的苛求**：在线调用大模型 API 存在不可控的网络延迟（几百毫秒到数秒），会严重拖死仿真时间轴。而正则表达式解析耗时是“微秒级”的。
   3.  **指令本质是“受限语言”**：飞行简令包含的要素高度结构化，正则表达式足以完美且稳定地胜任信息抽取工作。

   **未来演进预案**：
   如果在 V2.0 阶段，业务强烈要求支持像“*派两架隐形战机九点从大兴走，中午到上海*”这种极度口语化的自由文本，**我们的应对策略依然不是调用在线大模型，而是：利用标注数据，在本地部署一个极小参数的开源模型（如 1.5B 级别），专门针对“文本转JSON”做微调（LoRA）。** 既拥有理解口语的能力，又把延迟控制在本地毫秒级，彻底隔绝外网依赖。

   ## 7. 工程落地建议 (MVP 阶段)

   不要在第一阶段就去搞民航局真实的、包含复杂高度层和侧向偏移的庞杂航路数据库，工程量过大且容易陷入细节。

   **起步建议**：
   1.  用 Python 在地图上（比如大兴到虹桥之间）**人工点选 5~8 个坐标**，存成一条假的“京沪航线”数组。
   2.  把这 5~8 个坐标作为数组装进代码里，实现“航路展开”逻辑。
   3.  重点把“**圆弧平滑转弯**”的算法写进去。
   4.  只要跑起来后，在态势图上看到飞机的轨迹不再是生硬的折线，而是带有优美弧度的平滑曲线，且 `heading_deg` 数据连续变化，这个“高保真”的核心目的就达到了。后续再慢慢往数据库里灌真实的航路点数据即可。
