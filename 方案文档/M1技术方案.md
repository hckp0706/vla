# M1：意图驱动航迹生成器 技术方案

## 1. 模块定位与功能概述

在整个雷达组网仿真系统中，M1 扮演着**“战场导演”**的角色。它的核心任务是：接收指挥员下达的飞行任务指令，结合真实的地理环境、空域航路规则和飞机性能参数，推演出一条符合物理规律与航空逻辑的“上帝视角”飞行轨迹。

M1 生成的数据是整个系统数据流的源头。它不关心雷达能不能看到目标，它只负责如实地模拟出飞机在每一秒（1Hz频率）的位置、速度、姿态以及电磁反射特征（RCS），并将这些数据打包向后传递给 M2（雷达仿真引擎）。

---

## 2. 数据流转与处理步骤

对于刚接触本系统的工程师，可以将 M1 内部的工作流理解为一条“加工流水线”，主要分为四个步骤：

1.  **文本解析（提取要素）**：将人类读得懂的指令文本，拆解为程序能识别的“批号、时间、地点、动作”等孤立要素。
2.  **知识映射（补全信息）**：拿着解析出的“地名”和“机型”，去本地知识库查表，把地名变成经纬度，把机型变成速度、高度限制参数；**同时根据机型匹配对应的预设航路，将起降两点展开为详细的航路点序列**。
3.  **物理推演（生成轨迹）**：根据航路点序列和时间要求，利用地球椭球模型计算大圆航线，并处理航路点之间的平滑转弯，生成每秒的经纬度、速度和高度。
4.  **特征估算（计算RCS）**：根据飞机当前的飞行状态（如是否放起落架、是否在急转弯），估算出当前时刻的雷达散射截面积（RCS），一并写入最终的数据包。

class EAIPDataLoader:

        with open('data/eaip_airports.json', 'r', encoding='utf-8') as f:

        return {a['icao_code']: a for a in airports}

    def _load_waypoints(self):
        """加载航路点数据"""
        with open('data/enr_4_4_waypoints.json', 'r', encoding='utf-8') as f:

        """根据名称获取航路点信息"""
        return self.waypoints.get(name)

### 4.2 航路选择算法

```python
def plan_route(start_code, end_code, waypoints_db):
    """
    使用EAIP数据规划航线

    算法步骤：
    1. 获取起终点坐标
    2. 在航路点数据库中筛选候选点
    3. 使用A*或动态规划选择最优路径
    4. 构建完整航路点序列
    """
    # 1. 获取起终点
    start = waypoints_db.get_airport(start_code)
    end = waypoints_db.get_airport(end_code)

    # 2. 筛选候选航路点（根据距离和航路归属）
    candidates = filter_waypoints(start, end, waypoints_db.waypoints)

    # 3. 选择最优路径
    optimal_route = find_optimal_path(candidates, start, end)

    # 4. 返回完整航路
    return [start_code] + optimal_route + [end_code]
```

### 6.3 本地代码核心计算逻辑（时间与轨迹）

#### 时间计算逻辑

本地代码负责根据机型性能参数和航路距离，计算途经点时间和降落时间。

**时间计算公式**：

| 飞行阶段 | 速度来源 | 计算公式 |
|----------|----------|----------|
| 起飞阶段 | 机型参数 | 时间 = 起飞滑跑距离 / 平均起飞速度 |
| 爬升阶段 | 机型参数 | 时间 = (巡航高度 - 当前高度) / 爬升率 |
| 巡航阶段 | 机型参数 | 时间 = 航段距离 / 巡航速度 |
| 下降阶段 | 机型参数 | 时间 = (当前高度 - 降落高度) / 下降率 |
| 降落阶段 | 机型参数 | 时间 = 降落滑跑距离 / 平均降落速度 |

**示例计算**（北京-青岛-上海航线）：

## 7. 核心算法与物理逻辑详解

### 7.1 文本解析逻辑（正则提取）

```python
import re

# 定义匹配规则，将简令拆解为多个命名组
INTENT_PATTERN = re.compile(
    r"(?P<target_id>\d+)\s+(?P<platform_type>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+于\s+"
    r"(?P<takeoff_time>\d{4}-\d{2}-\d{2}\s+\d{2}:\d{2}:\d{2})\s+从\s+"
    r"(?P<loc_start>[\u4e00-\u9fa5a-zA-Z0-9]+)\s+起飞"
    r"(?:，?\s*(?P<action_mid>[\u4e00-\u9fa5]+)\s+(?P<loc_mid>[\u4e00-\u9fa5a-zA-Z0-9]+))?"
    r"(?:，?\s*(?P<action_end>[\u4e00-\u9fa5]+)\s+(?P<loc_end>[\u4e00-\u9fa5a-zA-Z0-9]+))?"
)
```

### 7.2 航路展开与平滑转弯算法

1.  **航路展开**：识别到“民航客机”后，程序自动去EAIP数据库匹配航路点，将原本的“大兴 -> 虹桥”两个端点，展开为包含十余个中间导航点的完整序列。
2.  **直飞段计算**：在两个相邻航路点之间，调用专业的地理空间库（如 `geographiclib`）计算大圆航线，正常进行 1Hz 的速度和经纬度插值。
3.  **转弯提前量计算**：飞机不会到了航路点正上方才生硬转弯。程序会根据当前速度和标准转弯率（通常为3度/秒），计算出**开始转弯的起点**和**结束转弯的终点**。
4.  **圆弧平滑插值**：在转弯区间内，程序以航路点为圆心画一段圆弧。在这几秒内，飞机的 `heading_deg`（航向）会**平滑、匀速地**从上一个航向过渡到下一个航向，彻底消除航向突变。

### 7.3 航迹物理推演逻辑（高度与速度）

*   **宏观校验**：根据展开后的航路点序列计算总距离，除以总时间得出所需平均速度。若超出机型极限，按最大速度飞行并自动顺延到达时间。
*   **高度剖面**：模拟真实的“起飞拉起 -> 爬升 -> 巡航平飞 -> 下降 -> 落地”曲线。根据知识库的爬升/下降率，结合距离目标的远近，自动计算 `vertical_rate_ms` 的变化。

### 7.4 动态 RCS 估算逻辑

由于 M1 不知道雷达具体部署在哪里，采用基于“飞行姿态”的经验估算法则：
*   **巡航态**：平飞时取“机头RCS”作为基准（例如民航为 5.0 dBsm）。
*   **起降态增损**：起飞和降落阶段放下起落架，RCS = 基准值 + 起落架增量（如 5.0 + 8.0 = 13.0 dBsm）。
*   **转弯态增损**：如果在执行“圆弧平滑转弯”，说明侧面暴露。程序会将 RCS 从“机头值”向更高的“侧面值”进行线性插值过渡。

## 8. 异常处理与防呆机制

1.  **地名模糊匹配**：简令写“大兴”，知识库叫“北京大兴机场”，程序会自动计算字符串相似度进行模糊匹配修正，不直接报错。
2.  **地形防穿透**：若接入高程地形数据（DEM），每秒生成高度点时校验，若低于地表安全高度，强制拉平，防止出现“飞机钻地”。
3.  **超时柔性处理**：遇到物理上飞不到的任务，不抛出异常中断仿真，而是按极限性能飞，并记录警告日志。
4.  **EAIP数据缺失处理**：若起终点不在EAIP数据库中，自动切换到大模型或本地预设航路。

## 9. 架构设计考量：EAIP为主、大模型为辅

### 9.1 为什么优先使用EAIP数据？

1.  **数据权威性**：EAIP是中国民航局发布的官方航空情报数据，具有最高的权威性和准确性。
2.  **真实性保障**：EAIP数据来源于实际航空运行，航路点、机场坐标等都是真实使用的。
3.  **稳定性保证**：本地数据不依赖网络，仿真过程稳定可靠。
4.  **成本效益**：无需支付API费用，降低运行成本。

### 9.2 大模型的辅助角色

在以下情况下，大模型作为补充手段：
- 起终点或途经点不在EAIP数据库中
- 需要生成非常规航线（如军机战术航线）
- 需要考虑实时气象因素调整航路

### 9.3 未来演进方向

1.  **持续更新EAIP数据**：随着民航局发布新版EAIP，定期更新本地数据库。
2.  **扩展数据覆盖**：补充更多中小型机场和通用航空机场数据。
3.  **集成气象数据**：接入气象服务，根据实时天气调整航路。
4.  **优化航路算法**：使用更智能的路径规划算法，提高航路选择的合理性。
