# M2传感器扩展方案 v2.0

## 第一部分：传感器调研

> 调研部分与v1.0一致，此处不再重复。核心结论：
> - **SSR**：1030/1090MHz 询问-应答，民航Mode 3/A+C+S，军机不响应
> - **IFF**：1030/1090MHz 加密识别，我方Mode 4/5正确应答→FRIEND，敌方不响应→FOE
> - **ESM**：被动截获0.5-18GHz雷达信号，输出PDW，侦察距离为雷达1.5-2倍
> - **ELINT**：基于ESM数据进行辐射源定位、指纹识别、威胁评估
> - **COMINT**：被动截获HF/VHF/UHF通信信号，网络分析

---

## 第二部分：架构设计

### 1. 核心架构决策

#### 1.1 模块归属：新传感器作为M2子模块

**决策：所有新传感器作为M2的子模块，不新建独立模块。**

理由：
- 项目目标是**雷达组网**，所有传感器服务于同一个目标——形成综合航迹
- 新传感器与现有雷达共享同一套输入（M1轨迹真值）和同一套融合引擎
- 保持项目模块边界清晰：M1负责轨迹生成，M2负责传感器仿真+融合，M4负责可视化
- 避免模块间循环依赖

#### 1.2 不修改现有雷达模块

**硬性约束：radar.py、models.py中的RadarConfig/TargetTruth/TrackPoint等现有类不做任何修改。**

实现方式：
- 新增数据模型放在 `models_ext.py`（扩展模型文件），不修改 `models.py`
- 新增传感器文件（ssr.py/iff.py/esm.py/elint.py/comint.py）独立实现
- 仿真引擎 `sensor_simulation.py` 仅做**增量扩展**，不改动现有雷达相关逻辑
- 现有 `test_m2.py` 必须继续全部通过

#### 1.3 综合航迹融合设计

**核心目标：所有传感器的观测信息融合成一条综合航迹。**

现有融合逻辑（sensor_simulation.py）：
- 多部雷达观测 → SNR加权平均位置 → 融合航迹（仅含位置、速度、航向、SNR、质量等级）

新融合设计：
- 综合航迹 = 位置信息（来自一次雷达+SSR） + 属性信息（来自SSR/IFF/ESM）
- 位置融合：一次雷达提供主位置，SSR Mode C/S提供高度和身份修正
- 属性融合：SSR提供身份/呼号，IFF提供敌我属性，ESM提供电磁特征

```
综合航迹数据结构：

FusedTrackPoint {

    time, track_id, target_id,
    obs_lon, obs_lat, obs_alt_m, obs_speed_ms, obs_heading_deg,
    track_quality, snr_avg_db, source_radars,

    # ---- 新增：属性信息（来自其他传感器）----
    identity: str,               # 敌我属性：FRIEND/FOE/UNKNOWN/NEUTRAL

```

### 2. 传感器部署设计

#### 2.1 部署原则

传感器不是随意放置的，需要考虑：
1. **与雷达共址**：SSR/IFF通常与一次雷达共址安装（同一天线座）
2. **独立部署**：ESM/COMINT可以独立部署在前沿或高地上
3. **几何约束**：交叉定位需要ESM站之间有足够基线长度
4. **覆盖互补**：传感器部署应弥补雷达盲区

#### 2.2 传感器与雷达站的关联关系

```
雷达站 RADAR_01 (青岛) ── 共址 ── SSR_01 + IFF_01
雷达站 RADAR_02 (上海) ── 共址 ── SSR_02 + IFF_02
雷达站 RADAR_03 (宁波) ── 共址 ── SSR_03
雷达站 RADAR_04 (南京) ── 共址 ── SSR_04 + IFF_04
雷达站 RADAR_05 (合肥) ── 共址 ── SSR_05 + IFF_05

ESM站 ESM_01 (舟山) ── 独立部署，前沿侦察
ESM站 ESM_02 (上海) ── 与RADAR_02共址
ESM站 ESM_03 (温州) ── 独立部署，南翼覆盖

ELINT处理节点 ELINT_01 ── 不需要物理位置，汇总ESM数据

COMINT站 COMINT_01 (舟山) ── 与ESM_01共址
COMINT站 COMINT_02 (上海) ── 与RADAR_02共址
```

#### 2.3 传感器可用性约束

不同场景下，传感器可能可用也可能不可用：

| 约束条件 | SSR | IFF | ESM | ELINT | COMINT |
|----------|-----|-----|-----|-------|--------|
| 目标为民航 | ✅ 可用 | ❌ 不适用 | ❌ 不适用 | ❌ 不适用 | ❌ 不适用 |
| 目标为我方军机 | ❌ 不应答 | ✅ 可用 | ❌ 不适用 | ❌ 不适用 | ❌ 不适用 |
| 目标为敌方军机 | ❌ 不应答 | ❌ 敌方不响应 | ✅ 可用 | ✅ 可用 | ✅ 可用 |
| 目标为未知目标 | ❌ 不应答 | ❌ 无响应 | ✅ 可用 | ✅ 可用 | ⚠️ 视情况 |
| 目标超视距 | ❌ 不可用 | ❌ 不可用 | ✅ 可用（距离更远） | ✅ 可用 | ⚠️ 视距离 |
| 目标在ESM覆盖外 | ❌ 视距离 | ❌ 视距离 | ❌ 不可用 | ❌ 无数据 | ❌ 不可用 |
| 无敌方雷达辐射 | - | - | ❌ 无信号 | ❌ 无数据 | - |
| ESM站数<2 | - | - | ✅ 测向 | ❌ 无法定位 | - |
| 加密不可用 | - | ⚠️ Mode4/5降级 | - | - | - |

**关键约束总结**：
- **SSR**：仅对配有应答机的民航/合作目标有效；需要目标在视距内；目标必须愿意应答
- **IFF**：仅对我方/友方军机有效；需要加密密钥同步；敌方永远不会正确应答
- **ESM**：需要敌方雷达开机辐射；被动传感器无法发现静默目标；至少2站才能交叉定位
- **ELINT**：依赖ESM数据，ESM不可用则ELINT无法工作
- **COMINT**：需要敌方通信发射信号；受通信距离和地形限制

### 3. 传感器误差、准确性、时效性模型

#### 3.1 误差模型

| 传感器 | 误差类型 | 误差值 | 说明 |
|--------|----------|--------|------|
| 一次雷达 | 测距误差 | σ = 100m（典型） | 高斯分布 |
| 一次雷达 | 方位角误差 | σ = 0.3° | 高斯分布 |
| 一次雷达 | 俯仰角误差 | σ = 0.3° | 高斯分布 |
| SSR Mode 3/A | Squawk码误差 | 0.1%误码率 | 窜扰导致 |
| SSR Mode C | 高度误差 | ±125ft（量化误差） | 100ft增量编码 |
| SSR Mode S | 位置误差 | 优于Mode C | 差分修正 |
| IFF Mode 4 | 识别错误率 | <0.1% | 加密验证 |
| IFF Mode 5 | 位置误差 | GPS精度（~10m） | 含GPS报告 |
| ESM | 测频误差 | σ = 1-5MHz | 取决于接收机 |
| ESM | 测向误差 | σ = 2-5° | 干涉仪测向 |
| ESM | PRI测量误差 | σ = 0.1μs | - |
| ELINT | 定位误差 | 2-10km（与基线相关） | 交叉定位 |
| ELINT | 识别准确率 | 80-95% | 取决于数据库完整度 |
| COMINT | 频率测量误差 | σ = 100kHz | - |
| COMINT | 方位测量误差 | σ = 3-5° | - |

#### 3.2 准确性模型

不同传感器对目标位置的测量准确性差异巨大：

```
位置准确性排序（从高到低）：
1. IFF Mode 5 位置报告  ~10m    （GPS级）
2. SSR Mode S            ~30m    （差分GPS）
3. 一次雷达（多站融合）  ~100m   （SNR加权）
4. SSR Mode C 高度       ~40m    （100ft量化）
5. 一次雷达（单站）      ~300m   （球坐标转换）
6. ELINT交叉定位         ~5km    （方位线交叉）
7. ESM单站测向           仅方位，无距离
8. COMINT                仅方位，无距离
```

**融合策略**：
- 位置融合：以一次雷达为主，SSR Mode C/S修正高度，IFF Mode 5修正位置
- 属性融合：IFF优先级最高（明确敌我），SSR次之（身份信息），ESM辅助（电磁特征）
- 准确性通过权重体现：精度高的传感器权重更大

#### 3.3 时效性模型

| 传感器 | 更新周期 | 延迟 | 说明 |
|--------|----------|------|------|
| 一次雷达 | 扫描周期（4-10s） | 实时 | 机械扫描，一个扫描周期更新一次 |
| SSR | 同一次雷达（共址） | 实时 | 与雷达同步扫描 |
| IFF | 同一次雷达（共址） | 实时 | 与雷达同步扫描 |
| ESM | 1-5s | 准实时 | 宽带接收，连续截获 |
| ELINT | 5-30s | 延迟处理 | 需要积累PDW数据后分析 |
| COMINT | 1-10s | 准实时 | 信号截获后分析 |

**时效性对融合的影响**：
- 雷达/SSR/IFF数据可直接融合（同一时间帧）
- ESM数据可近实时融合（1-5s延迟）
- ELINT数据需要时间积累，用于周期性更新属性而非实时位置
- COMINT数据用于补充属性，不参与位置融合

#### 3.4 融合时间窗口

```
时间轴示意（秒）：
t=0    t=4    t=6    t=8    t=10   t=12   t=14
 |      |      |      |      |      |      |
 R1─────┼──────R1─────┼──────R1─────┼──────R1     (10s周期)
 R2────┼────R2────┼────R2────┼────R2────┼────R2   (6s周期)
 S1─────┼──────S1─────┼──────S1─────┼──────S1     (SSR随R1)
 ESM─┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──             (连续截获)
 EL────────────────┬────────────────┬──             (30s周期)

融合窗口：±5s（现有逻辑）
- t=0~5s内的雷达+SSR+IFF+ESM观测 → 一帧融合航迹
- ELINT每30s更新一次属性
```

```python
        return json.dumps(self.to_dict(), ensure_ascii=False, indent=2)

    def save_to_file(self, file_path: str) -> None:
        with open(file_path, 'w', encoding='utf-8') as f:
            f.write(self.to_json())
```

## 第三部分：子模块详细设计

### 1. SSR二次雷达（ssr.py）

**职责**：仿真地面SSR的询问-应答过程

**输入**：
- M1轨迹数据（含ADS-B信息和has_adsb字段）
- SSRConfig配置
- 目标敌我属性（从sensor_config.json的target_identities获取）

**核心逻辑**：

```
对每个时间帧的每个目标：
1. 视距判定（复用LOS计算公式）
2. 距离判定（< max_range_km）
3. 应答资格判定：
   a. has_adsb=true → 民航，可应答SSR
   b. has_adsb=false 且 identity=FRIEND → 我方军机，不应答SSR
   c. has_adsb=false 且 identity=FOE → 敌方军机，不应答SSR
4. 对可应答目标：
   a. 应答概率判定（reply_probability）
   b. 旁瓣抑制判定（P1>P2）
   c. 按模式生成应答：
      - Mode 3/A: Squawk码（从ADS-B数据获取）
      - Mode C: 高度报告（从轨迹高度转换）
      - Mode S: ICAO24+呼号（从ADS-B数据获取）
   d. 异步干扰模拟（Fruit率）
5. 输出SSRReply列表
```

**不可用条件**：
- 目标为军机（has_adsb=false）→ 不应答
- 目标超视距 → 无法询问
- 目标超最大距离 → 无法询问

### 2. IFF敌我识别（iff.py）

**职责**：仿真IFF询问-应答过程

**输入**：
- M1轨迹数据
- IFFConfig配置
- 目标敌我属性

**核心逻辑**：

```
对每个时间帧的每个目标：
1. 视距判定
2. 距离判定
3. IFF询问模式选择（Mode 5优先，Mode 4备选）
4. 应答判定：
   a. identity=FRIEND → 正确应答
      - Mode 4: 加密验证通过 → FRIEND, confidence=0.90
      - Mode 5: 加密验证+GPS位置 → FRIEND, confidence=0.95
   b. identity=FOE → 不应答或错误应答 → FOE, confidence=0.60
   c. identity=UNKNOWN → 无应答 → UNKNOWN, confidence=0.30
5. 加密不可用降级处理：
   - crypto_valid=false → Mode 4/5无法使用 → UNKNOWN
6. 输出IFFReply列表
```

**不可用条件**：
- 目标超视距 → 无法询问
- 目标超最大距离 → 无法询问
- 加密不可用 → IFF降级为不可用

### 3. ESM雷达侦察（esm.py）

**职责**：仿真ESM被动截获雷达信号

**输入**：
- 辐射源数据库（emitter_database.json中的敌方雷达）
- ESMConfig配置
- ESM与辐射源的几何关系

**核心逻辑**：

```
对每个时间帧的每个敌方辐射源：
1. 视距判定
2. 单向传播方程计算接收功率：
   Pr_dBm = Pt_dBm + Gt_dB(direction) + Gr_dB - 20*log10(4*π*R/λ) - L
3. 灵敏度判定（Pr > sensitivity_dbm）
4. 截获概率计算：
   - 雷达波束扫过ESM时的驻留时间
   - ESM扫描周期与雷达扫描周期的关系
5. PDW参数测量（加高斯误差）：
   - rf_ghz ± freq_accuracy_mhz
   - pw_us ± pw_accuracy_us
   - pri_us ± pri_accuracy_us
   - aoa_deg ± df_accuracy_deg
   - pa_dbm（计算值±1dB）
6. 辐射源类型初步识别：
   - 基于RF+PW+PRI与emitter_database匹配
   - 输出最可能类型和置信度
7. 输出PDWRecord列表
```

**不可用条件**：
- 辐射源未开机（emitter_database中active=false）
- 辐射源在ESM频段外
- 辐射源超视距
- 接收功率低于灵敏度

**与M1数据的关系**：
- ESM不直接消费M1轨迹数据
- ESM消费的是emitter_database中的敌方雷达参数
- 但ELINT的定位结果可以与M1中的敌方目标轨迹关联

### 4. ELINT电子情报（elint.py）

**职责**：基于ESM数据进行辐射源定位、识别、威胁评估

**输入**：
- ESM输出的PDWRecord列表（多站）
- ELINTConfig配置
- emitter_database.json（雷达指纹数据库）

**核心逻辑**：

```
每个analysis_period周期：
1. 收集各ESM站对同一辐射源的PDW数据
2. 多站交叉定位：
   a. 站数 >= min_stations_for_fix → 交叉定位
   b. 方位线交叉计算位置（最小二乘法）
   c. 计算定位误差（与基线长度和测向精度相关）
   d. 站数 < min_stations_for_fix → 仅测向，不定位
3. 辐射源识别：
   a. PDW特征统计（RF均值/方差，PRI模式等）
   b. 与emitter_database模板匹配
   c. 匹配置信度 > matching_threshold → 确认识别
4. 威胁评估：
   a. 基于radar_type确定基础威胁等级
   b. 基于距离修正（越近威胁越大）
   c. 基于工作模式修正（跟踪>搜索）
5. 工作模式分析（PRI变化模式）
6. 输出ELINTReport列表
```

**不可用条件**：
- ESM站数 < 2 → 无法定位（仍可识别）
- ESM无数据 → ELINT无法工作
- emitter_database无匹配 → 识别为UNKNOWN

### 5. COMINT通信情报（comint.py）

**职责**：仿真通信信号截获和分析

**输入**：
- 通信辐射源参数（emitter_database.json中的comm_emitters）
- COMINTConfig配置

**核心逻辑**：

```
对每个时间帧的每个通信辐射源：
1. 视距判定
2. 频率范围判定（是否在COMINT覆盖频段内）
3. 接收功率计算
4. 截获判定（功率 > 灵敏度）
5. 通信参数测量（加误差）：
   - 频率 ± 100kHz
   - 方位 ± df_accuracy_deg
6. 通信网络分析：
   - 同一network_id的辐射源归入同一网络
   - 生成网络拓扑关系
7. 输出COMINTReport列表
```

**不可用条件**：
- 辐射源未发射
- 辐射源频率不在覆盖范围内
- 辐射源超视距
- 接收功率低于灵敏度

### 6. 综合融合引擎（fusion.py）

**职责**：将所有传感器数据融合成一条综合航迹

**输入**：
- 雷达观测数据（现有格式）
- SSR应答数据
- IFF识别数据
- ESM PDW数据
- ELINT报告数据
- COMINT报告数据

**核心逻辑**：

```
对每个融合时间窗口（±5s）：
1. 位置融合（现有逻辑不变）：
   - 多部雷达观测 → SNR加权平均位置
   - 质量等级判定（雷达数量）

2. 高度融合（新增）：
   - 优先使用SSR Mode C/S高度（精度更高）
   - 无SSR数据时使用雷达高度
   - 记录高度来源

3. 属性融合（新增）：
   a. 敌我属性融合：
      - IFF识别结果（最高优先级）
      - SSR应答推断（次高优先级）
      - 无数据时默认UNKNOWN
   b. 身份信息融合：
      - SSR Mode S提供ICAO24+呼号
      - 无SSR时留空
   c. 电磁特征融合：
      - ELINT报告中的辐射源类型和威胁等级
      - 关联到目标（基于方位/距离匹配）

4. 通信活动融合（新增）：
   - COMINT截获结果
   - 简化为有/无通信活动描述

5. 传感器贡献统计：
   - source_sensors: 所有贡献数据的传感器列表
   - sensor_count: 传感器数量

6. 输出FusedTrackPoint
```

**融合优先级规则**：

| 信息类型 | 优先级1 | 优先级2 | 优先级3 |
|----------|---------|---------|---------|
| 位置 | 多雷达融合 | 单雷达 | ESM定位 |
| 高度 | IFF Mode 5 | SSR Mode C/S | 雷达 |
| 敌我属性 | IFF Mode 5 | IFF Mode 4 | SSR推断 |
| 身份 | SSR Mode S | SSR Mode 3/A | 无 |
| 威胁等级 | ELINT评估 | ESM推断 | 无 |
| 通信活动 | COMINT截获 | 无 | 无 |
